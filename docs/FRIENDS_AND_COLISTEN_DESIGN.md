# Друзья и совместное прослушивание — дизайн и тестирование

## 1. Друзья (Friends)

### Модель данных

**Вариант A — только связи (проще)**  
Таблица `friends`: `user_id`, `friend_id`, уникальность по паре.  
Дружба двусторонняя: при добавлении можно создавать две строки (user→friend и friend→user) или считать «друзья» при запросе как `WHERE (user_id = ? AND friend_id = ?) OR (user_id = ? AND friend_id = ?)`.

**Вариант B — с заявками в друзья**  
- `friend_requests`: `from_user_id`, `to_user_id`, `status` (pending / accepted / rejected), `created_at`.  
- После `accepted` можно дублировать связь в таблицу `friends` для быстрых выборок «список друзей».

Рекомендация: начать с варианта A; при необходимости потом добавить `friend_requests`.

### API (REST)

- `POST /friends` — добавить друга (body: `{ "friendId": 123 }`), проверять что пользователь авторизован (токен → userId).
- `DELETE /friends/{friendId}` — удалить из друзей.
- `GET /friends` — список друзей текущего пользователя (по токену).
- Опционально: `GET /users/{id}/friends` — список друзей пользователя (если нужен публичный профиль).

Важно: везде брать `userId` из сессии/токена, а не из тела запроса, иначе безопасность ломается.

### Тестирование друзей

1. **Unit-тесты** логики (например, «добавление друга», «не добавить себя», «идемпотентность»).
2. **Интеграционные тесты** через `testApplication`:  
   - завести двух пользователей (или мокать БД);  
   - от имени одного вызвать `POST /friends` с `friendId` второго;  
   - `GET /friends` и проверить, что второй есть в списке;  
   - `DELETE /friends/{id}` и снова `GET /friends`.
3. **БД в тестах**: использовать H2 in-memory (как в существующих тестах), в `DatabaseFactory.init()` подставлять тестовый URL при `io.ktor.development=true` или отдельном флаге.

## 2. Совместное прослушивание (WebSockets)

### Зачем WebSockets

Для совместного прослушивания нужна **синхронизация в реальном времени**: play/pause, смена трека, seek. HTTP запросы для этого неудобны (поллинг тяжёлый и с задержкой). WebSocket даёт постоянное соединение и мгновенную рассылку событий всем в «комнате».

### Модель: комнаты (rooms)

- **Комната** — одна «сессия» совместного прослушивания.  
  Поля: `roomId`, `ownerId`, список `participantIds`, `currentTrackId`, `positionSeconds`, `playing: Boolean`, возможно `createdAt`.
- **Участники** подключаются по WebSocket к эндпоинту вида `/ws/room/{roomId}` (или `/ws/colisten/{roomId}`).
- **События** (примеры):
  - **От клиента к серверу**: `join`, `leave`, `play`, `pause`, `seek(positionSec)`, `changeTrack(trackId)`.
  - **От сервера к клиентам**: те же события + `user_joined`, `user_left`, чтобы все в комнате видели одно и то же состояние и список участников.

Хранить комнаты можно **в памяти** (Map по `roomId`), чтобы не усложнять первый вариант. При рестарте сервера комнаты пропадут — это приемлемо для MVP. Позже можно персистить в БД (например, только метаданные комнаты и последнее состояние).

### Протокол по WebSocket

Текстовые JSON-фреймы, например:

```json
{ "type": "play" }
{ "type": "pause" }
{ "type": "seek", "position": 45.2 }
{ "type": "change_track", "trackId": 7 }
{ "type": "state", "trackId": 7, "position": 12.5, "playing": true, "participants": [1, 2, 3] }
```

Сервер при любом изменении рассылает всем в комнате актуальное `state` (или отдельные события).

### Авторизация WebSocket

- Вариант 1: токен в query, например `/ws/room/abc-123?token=...`. При подключении проверять токен, доставать `userId`, добавлять в комнату.
- Вариант 2: первый текстовый фрейм после подключения — `{"type":"auth","token":"..."}`. Сервер проверяет и только после этого считает участника «в комнате».

Рекомендация: query с токеном проще и сразу отсекает неавторизованных.

### Ограничение: только друзья

- Создание комнаты: только авторизованный пользователь; в метаданные комнаты записать `ownerId`.
- Вход в комнату: проверять, что `userId` участника есть в списке друзей владельца (или что владелец в друзьях у участника). Тогда совместное прослушивание только с друзьями.

### Тестирование WebSocket

1. **Ktor TestEngine** поддерживает WebSocket на клиенте: в `testApplication` можно открыть `webSocket("/ws/room/...")` и писать/читать фреймы.
2. Сценарии:
   - Два «клиента» подключаются к одной комнате (с разными токенами двух пользователей).
   - Один шлёт `play`, второй получает событие (или обновлённое `state`).
   - Один шлёт `seek`, второй получает новую позицию.
3. Мокать или реально создавать пользователей и комнату в памяти; при необходимости мокать только слой «друзья» (всегда возвращать «друг»), чтобы не поднимать полную БД в тесте.

Имеет смысл вынести «менеджер комнат» в отдельный сервис/объект с методами `createRoom(ownerId)`, `joinRoom(roomId, userId)`, `leaveRoom(roomId, userId)`, `broadcastState(roomId)`, чтобы его тестировать отдельно unit-тестами (без реального WebSocket).

### Тесты в проекте

- **ColistenRoomManagerTest** — unit-тест менеджера комнат (create/join/broadcast/leave) без HTTP и БД.
- **FriendsRoutingTest** — вызов `GET /friends` без токена и с невалидным токеном → 401. Полные сценарии (добавление друга, список) требуют БД: в `testApplication` можно вызвать `DatabaseFactory.init()` с H2 (jdbcUrl = "jdbc:h2:mem:test;DB_CLOSE_DELAY=-1") и создать пользователей/токены перед тестами.
- **WebSocket**: в `testApplication` можно использовать `client.webSocket("/ws/room/...") { ... }` для проверки подключения и обмена фреймами (при необходимости добавить зависимость `ktor-client-websockets` в тестовый scope).

## 4. Как тестировать вручную

- **REST**: Postman/Insomnia/curl — логин, токен в заголовке, вызовы `/friends`, `/tracks`, `/search`.
- **WebSocket**:  
  - Postman или браузер (например, простой HTML с `new WebSocket(...)` и выводом в консоль).  
  - Подключиться к `/ws/room/{roomId}?token=...`, слать JSON-сообщения, смотреть ответы.  
  - Два окна/два пользователя — убедиться, что play в одном окне отражается во втором.

После этого можно добавить минимальный фронт (или расширить существующий) для создания комнаты, приглашения друзей и совместного прослушивания по WebSocket.

## 5. SQL для таблицы друзей

Если таблицы создаются вручную в PostgreSQL:

```sql
CREATE TABLE IF NOT EXISTS friends (
    user_id   INT NOT NULL REFERENCES users(iduser),
    friend_id INT NOT NULL REFERENCES users(iduser),
    PRIMARY KEY (user_id, friend_id),
    CHECK (user_id != friend_id)
);
CREATE INDEX IF NOT EXISTS idx_friends_friend_id ON friends(friend_id);
```
